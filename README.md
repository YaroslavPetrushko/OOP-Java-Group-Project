# 📚 Library Management System (JavaFX + JDBC)

## 📌 Опис проєкту

Цей проєкт є десктопним GUI-застосунком для роботи з базою даних бібліотеки.  
Метою є реалізація CRUD-операцій та пошуку для управління книгами, авторами та читачами.

Застосунок дозволяє:
- додавати, редагувати та видаляти записи;
- переглядати інформацію у вигляді таблиць;
- виконувати пошук за різними критеріями;
- працювати з базою даних PostgreSQL через JDBC.

---

## 🧱 Використані технології

- **Java 25**
- **JavaFX 25** — графічний інтерфейс
- **JDBC** — взаємодія з БД
- **PostgreSQL 18** — система управління базою даних
- **Maven** — управління залежностями
- **MVC** — архітектурний шаблон

---

## 🏗 Архітектура застосунку

Проєкт побудований за шаблоном **MVC**:

- **Model** — класи предметної області + DAO
- **View** — JavaFX (FXML)
- **Controller** — обробка подій користувача
- **Service/DB layer** — взаємодія з БД через JDBC

Всі SQL-запити реалізовані через **PreparedStatement**.

## 🗄 Схема бази даних

### Основні таблиці:

#### 📖 authors
- id
- name

#### 📚 books
- id
- title
- author_id (FK)
- isbn
- genre

#### 👤 readers
- id
- name
- email

#### 🔄 loans
- id
- book_id (FK)
- reader_id (FK)
- borrow_date
- return_date

### Зв’язки:

- authors **1 → N** books  
- books **1 → N** loans  
- readers **1 → N** loans  

> Таблиця `loans` реалізує зв’язок багато-до-багатьох між книгами та читачами.

## 🌿 Робота з Git

### 🔹 Структура гілок

- `main` — стабільна версія
- `develop` — основна гілка розробки
- `feature/*` — новий функціонал
- `fix/*` — виправлення помилок

---

### 🔹 Правила комітів

Формат:
```
type: короткий опис
```
#### Типи комітів:

- `feat` — новий функціонал
- `fix` — виправлення помилки
- `refactor` — рефакторинг
- `docs` — документація
- `style` — форматування
- `test` — тести

#### Приклади:
```
 feat: add book search functionality
 fix: correct SQL query in BookDAO
 refactor: improve DB connection handling
```
---

### 🔹 Pull Request правила

- PR створюється у `develop`
- мінімум 1 рев’ю (Team Lead)
- заборонено пушити напряму в `main`
- кожен PR повинен мати:
 - опис змін
 - пояснення логіки
